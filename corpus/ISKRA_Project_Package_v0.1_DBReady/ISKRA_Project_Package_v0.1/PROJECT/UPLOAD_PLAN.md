# UPLOAD_PLAN — как грузить файлы в Project (v0.1)

## Почему нужен план
Большие документы лучше грузить частями, чтобы:
- не упереться в лимиты загрузки/обработки
- улучшить поиск по знаниям (меньше «шумного» контента)
- ускорить ответы (меньше «лишнего» контекста)

## Рекомендованный порядок (практичный)
### Шаг 1 — «опорный набор» (10 минут)
1) `PROJECT/INDEX.md`
2) `PROJECT/GOVERNANCE.md`
3) `PROJECT/DECISION_LOG_TEMPLATE.md`
4) `PROJECT/RESEARCH_PROTOCOL_SIFT.md`
5) `PROJECT/EVALS_PLAYBOOK_MINI.md`
6) `PROJECT/CHAT_STARTERS.md`

### Шаг 2 — Канон v5 (ядро поведения и терминов)
- `KNOWLEDGE/CANON_v5/*` (00–15 + отчёты)

### Шаг 3 — Справочные документы из архива
- `SOURCE_DOCS/*`

### Шаг 4 — Данные (порциями)
- `INGEST_SPLITS/dialogs/*` (лучше помесячно/по датам)
- `INGEST_SPLITS/shadow_logs/*` (частями)

### Шаг 5 — Исследовательские материалы (по необходимости)
- `KNOWLEDGE/RESEARCH_COG_AI/*`

## Если что-то «не лезет»
- Грузи порциями (по месяцам / по темам)
- Убирай дубликаты (см. `ARCHIVE_RESEARCH/DUPLICATES_REPORT.md`)
- Держи отдельный «архивный проект» для тяжёлых логов (если нужно)

## Правило простоты
В проекте должен быть **только контент, который реально помогает** решать текущие задачи.
